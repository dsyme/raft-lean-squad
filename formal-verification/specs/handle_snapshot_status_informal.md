## Purpose

`handle_snapshot_status` is the leader-side handler for `MsgSnapStatus` messages,
which report the outcome of an in-flight snapshot send to a follower.

When a leader is sending a snapshot to a stalled follower, the follower's progress
entry is set to `ProgressState::Snapshot` with `pending_snapshot` pointing to the
snapshot index. Once the snapshot transfer completes (or fails), the follower (or
transport layer) sends back a `MsgSnapStatus` message (`reject = false` for success,
`reject = true` for failure).

This function transitions the follower's progress from `Snapshot` back to `Probe`
so that the leader can resume normal `AppendEntries` replication.

---

## Relevant Rust code

```rust
fn handle_snapshot_status(&mut self, m: &Message) {
    let pr = match self.prs.get_mut(m.from) {
        Some(pr) => pr,
        None => { return; }  // unknown peer — ignore
    };
    if pr.state != ProgressState::Snapshot {
        return;  // stale status — peer not in snapshot state
    }
    if m.reject {
        pr.snapshot_failure();   // pending_snapshot = 0
        pr.become_probe();       // Snapshot → Probe, next_idx = max(matched+1, 1)
    } else {
        pr.become_probe();       // Snapshot → Probe, next_idx = max(matched+1, ps+1)
    }
    pr.pause();                                    // paused = true
    pr.pending_request_snapshot = INVALID_INDEX;   // = 0
}
```

`snapshot_failure()` sets `pending_snapshot = 0`.
`become_probe()` from Snapshot state:
- reads `pending_snapshot`
- calls `reset_state(Probe)` (sets state = Probe, pending_snapshot = 0, paused = false)
- sets `next_idx = max(matched+1, old_pending_snapshot+1)`

---

## Preconditions

- Called by a leader node only.
- `m.from` identifies a known follower.
- The follower's progress entry is in `ProgressState::Snapshot` (otherwise ignored).
- `m.reject` is true if the snapshot transfer failed, false if it succeeded.
- `pending_snapshot` holds the index of the snapshot that was being sent.

---

## Postconditions

1. **Guard: unknown peer** — If `m.from` is not in the progress tracker, the function
   returns immediately. No state change.

2. **Guard: not in Snapshot state** — If the peer's progress state is not Snapshot,
   the function returns immediately. No state change.

3. **Both paths: state becomes Probe** — After the function, the progress state is
   always `ProgressState::Probe`.

4. **Both paths: paused = true** — After `become_probe` clears `paused`, the
   explicit `pr.pause()` call sets it back to true. The leader waits for an
   `AppendResponse` (or heartbeat timeout on failure) before sending again.

5. **Both paths: pending_snapshot = 0** — `reset_state` (called inside `become_probe`)
   always clears `pending_snapshot`.

6. **Both paths: pending_request_snapshot = INVALID_INDEX (0)** — Reset explicitly at
   the end.

7. **On success (`reject = false`)**:
   - `next_idx = max(matched + 1, old_pending_snapshot + 1)`
   - The leader probes from just after the snapshot, so the follower can confirm the
     snapshot and proceed from there.

8. **On failure (`reject = true`)**:
   - `pending_snapshot` is cleared to 0 before `become_probe`.
   - `next_idx = matched + 1` (since `max(matched+1, 0+1) = matched+1`).
   - The leader retries sending a snapshot after the next heartbeat.

9. **`matched` is unchanged** — This function never advances `matched`.

---

## Invariants

- **`next_idx ≥ matched + 1`** (the basic progress invariant) is preserved in both paths.
- **Monotone next_idx on success**: `next_idx ≥ old_pending_snapshot + 1` on success path.
- **Proof of state convergence**: After the call, state = Probe, regardless of input.

---

## Edge Cases

- **Stale MsgSnapStatus**: If progress state has already transitioned away from Snapshot
  (e.g., by a concurrent `MaybeUpdate`), the function ignores the message. This prevents
  double-transitions.
- **Snapshot failure with large pending_snapshot**: Even if `pending_snapshot` was large,
  clearing it first means `next_idx` is reset to `matched + 1`, not `pending_snapshot + 1`.
  This is correct: on failure we don't know how much the follower received.
- **Zero matched**: If `matched = 0`, `next_idx` is set to 1 on failure; on success it's
  `max(1, pending_snapshot+1) = pending_snapshot+1`.

---

## Examples

**Success, pending_snapshot = 100, matched = 50**:
- `next_idx = max(51, 101) = 101`
- state = Probe, paused = true

**Failure, pending_snapshot = 100, matched = 50**:
- `snapshot_failure()` → `pending_snapshot = 0`
- `next_idx = max(51, 1) = 51`
- state = Probe, paused = true
